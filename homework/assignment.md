# Простой MCP-сервер

## Цель

собрать и запустить собственный MCP‑сервер, который предоставляет агенту набор кастомных инструментов (tools), и показать интеграцию с агентом в IDE, чтобы вызовы инструментов реально происходили из IDE‑чата/агента;

## Описание/Пошаговая инструкция выполнения домашнего задания

Что нужно сделать:

1. Подготовьте проект и окружение
    - Создайте минимальный проект (отдельная папка в существующем репозитории допустима).
    - Зафиксируйте зависимости (package.json/pnpm-lock.yaml или requirements.txt/pyproject.toml).
    - Секреты и ключи не коммитить. Добавьте .env.example и описание настройки.
2. Разберитесь в принципах MCP
    - Коротко опишите в отчёте (1–2 абзаца):
        - как IDE/агент подключается к MCP‑серверу;
        - что именно вы считаете «tool» в вашем сервере.
3. Реализуйте MCP‑сервер
    - Сервер должен поднимать MCP‑эндпоинт (по стандартному транспорту, который выбран вашей IDE/клиентом: stdio или HTTP — в зависимости от выбранного SDK и способа подключения).
    - Сервер должен объявлять минимум 2–4 инструмента.
    - Каждый инструмент:
        - имеет понятное имя, описание и схему входных параметров;
        - возвращает структурированный результат (JSON/объект), а не «просто текст».
4. Сделайте инструменты полезными (ориентир на Context7‑подход)
    Выберите один из практичных вариантов (или свой):
    - Doc/Code context tool: инструмент, который по запросу (package/class/function) возвращает справку/выжимку/ссылки на документацию из локального источника (например, markdown в репо, подготовленный вами).
    - Project helper tool: чтение/поиск по локальным файлам проекта (например: «найди места использования X», «покажи публичные методы Y»), с ограничениями доступа (только папка проекта).
    - DevOps/tooling tool: выполнение безопасных команд (например, npm test/pytest/линтер) с whitelisting.
    Требование безопасности:
    - если инструмент выполняет команды/читает файлы — ограничьте область и явно опишите ограничения в README.
5. Интегрируйте MCP‑сервер с агентом в IDE (обязательное требование)
    - Подключите MCP‑сервер в вашей IDE (например, через конфиг MCP‑серверов, marketplace‑плагин, настройки агента и т.п.).
    - В репозитории должен быть:
    - пример конфигурации подключения (например, mcp.json, settings.json, cursor/mcp.json, и т.д. — в зависимости от IDE);
    - инструкция «как включить» (2–5 шагов).
6. Проверка: продемонстрируйте вызовы инструментов из IDE
    - Выполните минимум неесколько запросов к агенту в IDE, м 3 запроса должны приводить к реальному вызову MCP‑tool.
    - Зафиксируйте:
        - запрос пользователя;
        - какой инструмент должен быть вызван (ожидаемо);
        - фактическое подтверждение вызова (лог/trace/скриншот/вывод).
7. Логи и отладка (обязательное требование)
    - Добавьте вывод (логгер/print) на стороне MCP‑сервера, который явно показывает:
        - какое имя инструмента вызвали;
        - какие входные параметры пришли (без секретов);
        - какой статус выполнения (success/error).

## Критерии оценки

Работа считается выполненной, если:

1. MCP‑сервер запускается по инструкции из репозитория.
2. Реализованы минимум 2 инструмента (лучше 3–4) с описаниями и параметрами.
3. Интеграция с агентом в IDE показана и воспроизводима по шагам.
4. Есть минимум 5 проверочных запросов; минимум 3 приводят к вызову MCP‑tool.
5. Добавлены логи/отладочный вывод на стороне сервера.
6. Секреты не закоммичены.

Подтверждения ссылками на код (обязательно)

В отчёте/README добавьте пункты с ссылками на файлы и диапазоны строк:

1. Реализован MCP‑сервер — укажите файл и диапазон строк, где поднимается сервер и происходит регистрация инструментов (например: server.ts:L10–L120).
2. Реализован инструмент(ы) — для каждого tool укажите:
    - файл и диапазон строк с реализацией;
    - где добавлен вывод в консоль/логи для дебага (файл/строки);
    - пример вывода (фрагмент).
3. Агент корректно вызывает нужный tool — приведите минимум 1 пример запроса, который должен вызвать конкретный tool, и укажите:
    - ожидаемый tool;
    - фактическое подтверждение (лог/trace/скриншот) + где это хранится в репозитории.
4. Требования к формату результатов tool/ответа — укажите, где описан контракт результата (например: README.md: раздел “Tool outputs contract” или docs/contract.md:L1–L60).
