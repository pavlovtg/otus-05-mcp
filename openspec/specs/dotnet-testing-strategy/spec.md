## ADDED Requirements

### Requirement: Три уровня тестирования

Каждый .NET сервис в репозитории SHALL покрываться тестами трёх уровней в одном тестовом проекте:

- **Unit-тесты** — изолированное тестирование бизнес-логики без внешних зависимостей
- **Integration-тесты** — тестирование взаимодействия компонентов с реальными зависимостями (файловая система, БД и др.)
- **API-тесты** — полный вызов HTTP эндпоинтов сервиса (микросервисные тесты)

#### Scenario: Наличие тестового проекта

- **WHEN** разработчик создаёт новый .NET сервис
- **THEN** в `apps/<app>/tests/` присутствует один тестовый проект `<ProjectName>.Tests`, содержащий unit, integration и API-тесты

### Requirement: Минимальный порог покрытия кода

Code coverage .NET сервиса SHALL составлять не менее 80% по метрике line coverage.

#### Scenario: Проверка coverage

- **WHEN** запускаются тесты сервиса
- **THEN** итоговый line coverage составляет ≥ 80%

#### Scenario: Падение ниже порога

- **WHEN** coverage опускается ниже 80%
- **THEN** сборка считается неуспешной

### Requirement: Unit-тесты

Unit-тесты SHALL тестировать бизнес-логику изолированно, без обращения к файловой системе, сети или базе данных. Зависимости SHALL подменяться через моки/стабы.

#### Scenario: Тест бизнес-логики

- **WHEN** запускается unit-тест метода поиска контрактов
- **THEN** тест использует in-memory данные, не обращается к файловой системе

### Requirement: Integration-тесты

Integration-тесты SHALL проверять взаимодействие компонентов с реальными зависимостями. Для wiki-сервиса — чтение реальных файлов из временной директории.

#### Scenario: Тест чтения файлов

- **WHEN** запускается integration-тест чтения контрактов
- **THEN** тест создаёт временную директорию с тестовыми YAML-файлами и проверяет корректность чтения

### Requirement: API-тесты (микросервисные)

API-тесты SHALL выполнять полный HTTP-вызов к сервису. Для .NET сервисов SHALL использоваться `WebApplicationFactory<TProgram>` из `Microsoft.AspNetCore.Mvc.Testing`.

#### Scenario: Тест HTTP эндпоинта

- **WHEN** запускается API-тест эндпоинта `GET /api/contracts`
- **THEN** тест поднимает сервис через `WebApplicationFactory`, выполняет реальный HTTP-запрос и проверяет ответ

#### Scenario: Тест несуществующего контракта

- **WHEN** API-тест запрашивает `GET /api/contracts/nonexistent.yaml`
- **THEN** сервис возвращает `404 Not Found`

### Requirement: Расположение тестового проекта

Тестовый проект SHALL располагаться в `apps/<app>/tests/<ProjectName>.Tests/` согласно стандарту структуры .NET проектов.

#### Scenario: Структура тестового проекта wiki

- **WHEN** разработчик открывает `apps/wiki/tests/`
- **THEN** присутствует проект `Wiki.Api.Tests`, содержащий unit, integration и API-тесты в отдельных папках
