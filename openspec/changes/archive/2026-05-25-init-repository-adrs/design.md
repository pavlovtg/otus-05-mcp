## Контекст

- Репозиторий `otus-05-mcp` — учебный проект OTUS по разработке MCP-сервера на .NET
- Инициализация репозитория завершена (change `init-repository-rules`): правила Cline, memory bank, ADR-0001
- Отсутствуют зафиксированные решения по структуре директорий и code style
- Без ADR агент не имеет долгосрочного контекста по этим решениям

## Цели / Не-цели

**Цели:**

- Зафиксировать структуру директорий репозитория в ADR
- Зафиксировать базовые правила code style для C# и Markdown в ADR
- Привести шаблоны OpenSpec в соответствие с Markdown code style

**Не-цели:**

- Настройка линтеров и CI/CD для автоматической проверки code style
- Детальные правила архитектуры C# кода (паттерны, слои)
- Правила code style для других языков

## Решения

### Структура директорий

**Решение:** `/apps`, `/infrastructure`, `/docs`, `/homework` на верхнем уровне.

**Обоснование:** Разделение по назначению (код / инфраструктура / документация / учёба) упрощает навигацию. Альтернатива — плоская структура — не масштабируется при добавлении новых компонентов.

**Альтернативы:**

- `src/` вместо `apps/` — менее явно для проекта с несколькими приложениями
- Монорепо с `packages/` — избыточно для учебного проекта

### Code style C#: табы

**Решение:** Табы как символ отступа.

**Обоснование:** Табы позволяют каждому разработчику настроить визуальную ширину отступа в редакторе. Стандарт .NET SDK по умолчанию использует пробелы, но для учебного проекта выбор таба — осознанное решение команды.

**Альтернативы:**

- 4 пробела (стандарт Microsoft) — не выбрано, т.к. команда предпочитает табы

### Code style Markdown: markdownlint правила

**Решение:** Зафиксировать 6 правил markdownlint (MD009, MD012, MD022, MD032, MD034, MD047).

**Обоснование:** Эти правила обеспечивают совместимость с большинством Markdown-парсеров и читаемость в raw-виде. Правила выбраны как минимально необходимый набор без избыточных ограничений.

**Альтернативы:**

- Полный набор markdownlint — избыточно, создаёт трение при написании документации
- Без правил — приводит к несовместимым форматам между файлами

### Обновление шаблонов OpenSpec

**Решение:** Привести все 5 шаблонов (`proposal.md`, `design.md`, `spec.md`, `adr.md`, `tasks.md`) в соответствие с ADR-0004.

**Обоснование:** Шаблоны — источник новых файлов; если они нарушают code style, все создаваемые артефакты будут нарушать его тоже.

## Риски / Компромиссы

- [Риск] Существующие MD-файлы нарушают новые правила → Митигация: исправление шаблонов покрывает новые файлы; существующие файлы исправляются по мере работы с ними
- [Риск] Табы в C# конфликтуют с `.editorconfig` по умолчанию → Митигация: добавить `.editorconfig` с `indent_style = tab` в отдельном change

## План миграции

1. Создать ADR-файлы в `docs/adr/`
2. Обновить `docs/adr/README.md`
3. Обновить шаблоны OpenSpec

Откат не требуется — изменения аддитивны (новые файлы + обновление шаблонов).
