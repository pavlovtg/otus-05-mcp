## Контекст

- Репозиторий `otus-05-mcp` — учебный проект OTUS по разработке MCP-сервера на .NET
- В `apps/wiki/content/` накоплены swagger-контракты внутренних сервисов (YAML-файлы)
- Нет единой точки доступа к контрактам ни для людей, ни для программных потребителей
- Принятые ADR: ADR-0002 (структура директорий), ADR-0003 (C# code style), ADR-0004 (Markdown code style)
- ADR-0002 определяет `/apps` для приложений, `/infrastructure` для инфраструктуры, `/docs` для документации

## Цели / Не-цели

**Цели:**

- Предоставить web-интерфейс для просмотра swagger-контрактов (read-only)
- Предоставить REST API для программного доступа к каталогу контрактов (динамический)
- Зафиксировать стандарт структуры .NET проектов в репозитории
- Зафиксировать стандарт расположения контрактов сервисов (`docs/contracts/<service>/`)
- Зафиксировать стратегию тестирования .NET сервисов (три уровня, coverage ≥ 80%)
- Зафиксировать CI-подход (GitHub Actions, блокировка PR)
- Обеспечить развёртывание через Docker Compose одной командой

**Не-цели:**

- Выполнение HTTP-запросов к описываемым сервисам (только просмотр)
- Редактирование контрактов через UI
- Аутентификация и авторизация
- Динамическое обновление списка контрактов в Swagger UI без перезапуска контейнера

## Решения

### Стек: ASP.NET Core + Swagger UI

**Решение:** Backend — ASP.NET Core (C#, .NET 10), Frontend — официальный Docker-образ `swaggerapi/swagger-ui`.

**Обоснование:** ASP.NET Core соответствует ADR-0003 (C# как язык проекта). .NET 10 — актуальная версия, поддерживает `.slnx`. Swagger UI — стандартный инструмент для отображения OpenAPI/Swagger контрактов, не требует написания фронтенда. Два контейнера в docker-compose — минимальная сложность.

**Альтернативы:**

- .NET 8 LTS — стабильнее, но не поддерживает `.slnx` нативно
- ASP.NET Core с Razor Pages UI — требует написания фронтенда
- Redoc вместо Swagger UI — хорош для документации, но Swagger UI привычнее разработчикам

### Contract-first для wiki REST API

**Решение:** Swagger-контракт wiki REST API создаётся в `docs/contracts/wiki/wiki-api.yaml` до реализации.

**Обоснование:** Contract-first обеспечивает чёткий контракт для потребителей (MCP-сервер и др.) до начала реализации. Расположение в `docs/contracts/` соответствует новому стандарту хранения контрактов.

**Альтернативы:**

- Code-first (генерация swagger из кода) — контракт появляется после реализации, потребители не могут работать параллельно

### Swagger UI: конфигурация через URLS

**Решение:** Swagger UI настраивается через переменную окружения `URLS`, содержащую список URL эндпоинтов wiki REST API (`GET /api/contracts/{name}`) для каждого контракта. Список задаётся статически при старте docker-compose — при изменении набора файлов требуется перезапуск Swagger UI контейнера.

Формат `URLS`: `[{ url: "http://wiki-api:5000/api/contracts/File1.yaml", name: "File1" }, ...]`

**Обоснование:** Swagger UI поддерживает `URLS` env из коробки для отображения нескольких контрактов. Не требует кастомного кода. REST API при этом остаётся полностью динамическим для программных потребителей.

**Альтернативы:**

- Кастомный `index.html` с динамической загрузкой через `GET /api/contracts` — сложнее, избыточно для текущего масштаба

### REST API: динамическое чтение контрактов

**Решение:** Wiki REST API читает файлы из `CONTRACTS_PATH` при каждом запросе без кэширования. Список контрактов всегда актуален.

**Обоснование:** Обеспечивает hot-reload для программных потребителей (MCP-сервер) — добавление/удаление файлов отражается немедленно. Для учебного проекта с небольшим числом файлов производительность не критична.

**Альтернативы:**

- FileSystemWatcher + in-memory кэш — сложнее, избыточно для текущего масштаба

### Структура .NET проекта

**Решение:** `apps/wiki/src/` — исходники, `apps/wiki/tests/` — тесты, `apps/wiki/wiki.slnx` — solution.

**Обоснование:** Стандартная структура для .NET монорепо. Разделение `src/tests` упрощает настройку Docker (копируем только `src/`). `.slnx` — новый формат solution файла (XML-based, без GUID), поддерживается с .NET 9+.

**Альтернативы:**

- Плоская структура без `src/tests` — не масштабируется при добавлении нескольких проектов
- `.sln` вместо `.slnx` — устаревший формат

### Расположение контрактов сервисов

**Решение:** `docs/contracts/<service>/*.yaml` — стандарт для всех сервисов репозитория.

**Обоснование:** `docs/` — место для документации (ADR-0002). Отделение контрактов от данных wiki-сервиса (`apps/wiki/content/`) — разные назначения: `docs/contracts/` — контракты разрабатываемых сервисов репозитория, `apps/wiki/content/` — данные для wiki-сервиса (внешние контракты).

**Альтернативы:**

- Контракты рядом с кодом (`apps/<app>/contracts/`) — смешивает код и документацию

### Порты

**Решение:** Wiki REST API — порт `5000`, Swagger UI — порт `5100`.

**Обоснование:** Стандартный порт ASP.NET Core (5000). Swagger UI на соседнем порту (5100) для удобства.

### Имена Docker Compose проекта и контейнеров

**Решение:** Docker Compose проект называется `otus-05-mcp` (поле `name`). Контейнеры имеют фиксированные имена: `wiki-api` и `wiki-swagger-ui` (поле `container_name`).

**Обоснование:** Без фиксированных имён Docker генерирует имена на основе директории и случайного суффикса — это усложняет работу с контейнерами через CLI (`docker logs`, `docker exec`). Фиксированные имена делают команды предсказуемыми и документируемыми.

### CORS

**Решение:** Wiki REST API настраивает CORS политику `AllowAnyOrigin / AllowAnyMethod / AllowAnyHeader`.

**Обоснование:** Swagger UI работает на порту `5100` и делает browser-запросы к API на порту `5000` — разные origin. Без CORS браузер блокирует запросы. Для учебного проекта без аутентификации `AllowAnyOrigin` достаточно.

**Альтернативы:**

- Ограничить CORS конкретным origin (`http://localhost:5100`) — избыточно для учебного проекта, усложняет конфигурацию

### Dockerfile: сборка без --no-restore

**Решение:** `dotnet publish` в Dockerfile запускается без флага `--no-restore`.

**Обоснование:** Версия .NET SDK в Docker-образе может отличаться от локальной. При `--no-restore` publish использует `project.assets.json` от предыдущего restore, который может быть несовместим с версией SDK в контейнере — это приводит к ошибке `NETSDK1064`. Запуск publish без `--no-restore` гарантирует корректное восстановление пакетов внутри контейнера.

### Стратегия тестирования

**Решение:** Один тестовый проект `<ProjectName>.Tests` с тремя типами тестов (unit, integration, API). Code coverage ≥ 80%.

**Обоснование:** Один проект упрощает конфигурацию и запуск. Три типа тестов обеспечивают полное покрытие: unit — бизнес-логика, integration — файловая система, API — полный HTTP-стек через `WebApplicationFactory`.

**Альтернативы:**

- Отдельный проект на каждый тип тестов — избыточно для небольшого сервиса

### GitHub Actions CI

**Решение:** Один workflow `.github/workflows/dotnet-ci.yml`, запускающийся на `push` и `pull_request`. Шаги: `dotnet build` → `dotnet test` с coverage → проверка порога 80%. Branch protection rules блокируют слияние PR при падении.

**Обоснование:** GitHub Actions — встроенный CI для GitHub репозиториев, не требует внешних сервисов. Один workflow для всех .NET сервисов — единая точка конфигурации CI.

**Альтернативы:**

- Отдельный workflow на каждый сервис — избыточно, сложнее поддерживать

## Риски / Компромиссы

- [Риск] Swagger UI требует перезапуска при изменении набора контрактов → Принятый компромисс: статическая конфигурация через `URLS` достаточна для учебного проекта; REST API остаётся динамическим для программных потребителей
- [Риск] Path traversal при запросе файла по имени → Митигация: валидация имени файла (только имя без пути), проверка что результирующий путь находится внутри `CONTRACTS_PATH`
- [Риск] Coverage 80% может быть сложно достичь для API-слоя → Митигация: API-тесты через `WebApplicationFactory` покрывают контроллеры

## План миграции

1. Создать swagger-контракт `docs/contracts/wiki/wiki-api.yaml`
2. Создать структуру проекта `apps/wiki/` (src, tests, wiki.slnx)
3. Реализовать ASP.NET Core API согласно контракту
4. Написать тесты (`Wiki.Api.Tests`)
5. Создать `Dockerfile` для wiki-api
6. Создать `infrastructure/docker-compose/docker-compose.yaml` с `URLS` для Swagger UI
7. Создать `.github/workflows/dotnet-ci.yml`
8. Настроить branch protection rules в GitHub

Откат: удаление новых файлов, изменения аддитивны.
